# LangChain & LangGraph 완벽 가이드

LLM(대규모 언어 모델)을 기반으로 애플리케이션을 구축할 때 가장 널리 사용되는 두 프레임워크인 **LangChain**과 **LangGraph**에 대해 설명합니다.

---

## 1. LangChain 이란?

**LangChain**은 LLM을 활용한 애플리케이션 개발을 돕기 위한 오픈소스 프레임워크입니다. 여러 컴포넌트를 조합하여 기능을 "체인(Chain)"처럼 연결하는 것이 핵심입니다.

### 핵심 개념 (Core Components)
1. **Model I/O**: 다양한 LLM(GPT, Claude, Llama 등)과의 통신 및 프롬프트 관리.
2. **Retrieval (RAG)**: 외부 데이터(PDF, DB, 웹페이지 등)를 가져와 모델에 전달하는 기능.
3. **Chains**: 여러 동작(프롬프트 -> 모델 -> 파싱)을 하나의 작업 단위로 묶는 것.
4. **Memory**: 대화의 이전 내용을 기억하고 참조할 수 있게 하는 기능.
5. **Agents**: LLM이 스스로 판단하여 어떤 도구(검색, 계산 등)를 사용할지 결정하는 루프.

### 장점
- LLM 개발에 필요한 거의 모든 도구가 미리 구현되어 있어 생산성이 높습니다.
- 표준화된 인터페이스를 제공하여 모델 교체가 용이합니다.

---

## 2. LangGraph 란?

**LangGraph**는 LangChain 위에 구축된 라이브러리로, **"순환 구조(Cyclic)"**를 가진 복잡한 에이전트 시스템을 구축하기 위해 설계되었습니다.

### 왜 LangGraph가 필요한가?
기존 LangChain의 체인은 주로 선형적(Linear)인 흐름을 가집니다. 하지만 실제 비즈니스 로직이나 정교한 에이전트는 다음과 같은 특성이 필요합니다.
- **반복(Cycles)**: 결과가 만족스럽지 않으면 다시 이전 단계로 돌아가서 수정 요청.
- **상태 관리(State Management)**: 여러 단계에 걸쳐 복잡한 데이터를 안정적으로 유지.
- **세밀한 제어**: 에이전트의 판단 흐름을 그래프 형태로 직접 설계.

### 주요 요소
- **Nodes (노드)**: 특정 작업(파이썬 함수)을 수행하는 단계.
- **Edges (엣지)**: 노드 간의 이동 경로. (조건부 이동 가능)
- **State (상태)**: 모든 노드가 공유하고 업데이트할 수 있는 현재 데이터 스냅샷.

---

## 3. 에이전트(Agent)와 접목하기

### 에이전트(Agent)란?
에이전트는 LLM을 단순히 텍스트 생성 도구가 아닌 **"추론 엔진(Reasoning Engine)"**으로 사용하는 방식입니다. 사용자의 질문에 답하기 위해 어떤 도구(Tool)를 사용할지, 어떤 순서로 실행할지를 LLM이 스스로 판단하고 실행하는 시스템을 의미합니다.

### LangChain에서의 에이전트 (Legacy)
LangChain에서는 `AgentExecutor`라는 기성품 클래스를 사용하여 에이전트를 구현했습니다.
- **특징**: 빠르고 쉽게 에이전트를 만들 수 있지만, 내부 루프가 블랙박스처럼 작동하여 흐름을 세밀하게 수정하기 어렵습니다.
- **접목 방식**: `initialize_agent` 또는 최근의 `LangChain Expression Language (LCEL)`를 사용하여 도구 목록과 LLM을 묶어 실행합니다.

### LangGraph에서의 에이전트 (Modern)
LangGraph는 에이전트를 **상태가 있는 그래프(Stateful Graph)**로 정의합니다.
- **접목 방식**:
    1. **상태(State) 정의**: 에이전트가 주고받는 메시지 목록과 현재 데이터를 정의합니다.
    2. **노드(Node) 구성**: LLM이 판단하는 노드와 실제 도구를 실행하는 노드를 만듭니다.
    3. **조건부 엣지(Conditional Edge)**: LLM의 결과에 따라 "도구를 실행할 것인가(Continue)" 아니면 "최종 답변을 할 것인가(End)"를 결정하는 로직을 연결합니다.
- **장점**: 
    - 에이전트의 사고 과정을 단계별로 관찰하고 제어할 수 있습니다.
    - 에이전트가 루프에 빠졌을 때 강제로 탈출시키거나 사람이 중간에 개입(Human-in-the-loop)하는 것이 가능합니다.

---

## 4. LangChain vs LangGraph 한눈에 비교

| 구분 | LangChain (Chains) | LangGraph |
| :--- | :--- | :--- |
| **흐름 구조** | 선형적 (DAG: Directed Acyclic Graph) | 순환적 (Cyclic Graph) |
| **복잡성** | 단순한 워크플로우에 적합 | 복잡한 다중 에이전트 시스템에 적합 |
| **상태 유지** | 각 단계별 전달 방식 | 전역 상태(State) 기반 관리 |
| **유연성** | 정해진 순서대로 실행 | 조건에 따라 자유로운 이동 및 반복 가능 |

---

## 4. 어떤 것을 선택해야 할까요?

- **LangChain**을 선택하세요:
    - 단순한 질문-답변 시스템을 만들 때
    - 일반적인 RAG(검색 증강 생성) 구조를 구현할 때
    - 빠르게 프로토타입을 만들어야 할 때

- **LangGraph**를 선택하세요:
    - 에이전트가 스스로 수정하고 반복해야 하는 루프가 필요할 때
    - 여러 개의 에이전트가 협업하는 시스템(Multi-agent)을 구축할 때
    - 워크플로우의 특정 상태를 영구적으로 저장하고 복구(Checkpointing)해야 할 때

---

## 5. 요약

**LangChain**은 LLM 앱의 **레고 블록**과 같고, **LangGraph**는 그 블록들을 사용하여 복잡한 **기계의 두뇌(워크플로우)**를 설계하는 도구입니다. 처음 시작하신다면 LangChain의 기초를 익힌 후, 고도화된 에이전트가 필요할 때 LangGraph로 넘어가는 것을 추천합니다.
